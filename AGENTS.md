# bensz-devtools - 项目指令

本项目用于聚合多个软件的 vibe / devtools 远程桥梁 skill，并为后续新增产品保留一致的沉淀方式。

## 项目目标

- 把面向不同软件的远程配置类 skill 统一收纳在当前仓库的 `skills/` 下
- 保留每个 skill 的独立说明、脚本、配置、计划与测试沉淀
- 在不越界修改上游业务仓库的前提下，持续演进这些 bridge skill

本工作区的核心价值是把“多产品、同模式”的远程配置能力集中管理，避免分散在多个业务仓库里长期漂移。

## 当前范围

目前仓库内的核心 skill 有：

- `skills/dudu-vibe-config`：管理 `dudu` 的模板、报道风格、订阅、报道、域名规则
  - 源项目路径：`/Volumes/2T01/winE/Starup/dudu`
- `skills/bensz-channel-vibe-config`：管理 `bensz-channel` 的频道、标签、文章、评论、用户
  - 源项目路径：`/Volumes/2T01/winE/Starup/bensz-channel`

它们都属于“桥梁型 skill”：

- 输入是人类的配置或管理意图
- 中间层是本仓库里的 `scripts/` 命令行封装
- 输出是对目标系统受限 API 的调用

## 核心工作流

当用户提出本仓库相关需求时，按以下流程执行：

### 1. 任务理解

- 先确认目标是已有 skill 的维护、跨仓库迁移，还是新增一个产品的同类 skill
- 分清“修改当前 skill 仓库”和“修改上游产品代码”这两类任务，默认只做前者
- 如果用户提到具体 skill 名称，优先在对应目录内工作

### 2. 执行流程

审计现状 -> 确定边界 -> 修改当前仓库 -> 校验目录/脚本/文档一致性 -> 更新变更记录

### 3. 输出规范

- 优先引用实际文件路径，不粘贴大段文件全文
- 对迁移类任务，明确说明来源目录、保留内容和排除内容
- 对新增或修改的规则，说明它属于哪个 skill、影响哪个目标系统

## 仓库结构约定

每个 skill 原则上保持以下结构：

- `SKILL.md`：触发条件、目标、边界和操作流程
- `README.md`：面向人类的使用说明
- `config.yaml`：skill 的单一配置来源
- `scripts/`：可执行脚本和最小依赖的实现
- `docs/`：补充说明文档（推荐）
- `plans/`：设计或迭代计划（推荐）
- `tests/`：测试计划、报告和必要的验证工件（推荐）

其中 `SKILL.md`、`README.md`、`config.yaml`、`scripts/` 为核心资产；`docs/`、`plans/`、`tests/` 一旦存在，应尽量保留并纳入版本控制。允许保留产品特有差异，但不应引入无关缓存文件。

## 工程原则

本项目遵循以下原则：

| 原则 | 在本项目中的体现 |
|------|------------------|
| **KISS** | 优先保持每个 skill 自洽、易读，避免为了“统一”而过早抽象 |
| **YAGNI** | 只有当多个 skill 的重复模式稳定存在时，才考虑共享化 |
| **DRY** | 能复用的脚本模式要复用，但前提是不破坏产品边界 |
| **关注点分离** | skill 聚合仓库只管理 skill 资产，不承担上游业务代码开发 |
| **最小惊讶原则** | skill 名称、目录结构、脚本入口尽量一致，便于未来扩展到更多产品 |
| **安全优先** | 所有密钥都脱敏；默认不碰上游业务代码；只操作受限 API |

## 默认语言

除非用户明确要求其他语言，始终使用简体中文与用户对话、写说明和维护仓库文档。

## 联网与搜索

- 默认先使用当前仓库和来源仓库中的本地文件建立上下文
- 只有在需要核验外部 API、官方文档或最新行为时才联网
- 若任务只是迁移已有 skill，优先以本地源码和文档为准

## 变更边界

- 默认只修改当前仓库中的文件
- 除非用户明确要求，不修改 `/Volumes/2T01/winE/Starup/dudu` 或 `/Volumes/2T01/winE/Starup/bensz-channel` 的源文件
- 不擅自重命名现有 skill；如果要统一命名，先说明兼容性影响
- 不删除历史计划、测试和说明，除非它们是明显的缓存或垃圾文件

## 凭证管理

- `remote.env`：和bensz-channel、dudu交互所需的凭证文件，**绝对不能提交到版本控制**
- 该文件已由 `.gitignore` 中的 `*.env` 规则覆盖，但每次操作前须确认未被意外暂存
- 凭证仅在本地使用，不得硬编码到脚本或文档中

## 迁移与新增规则

当迁移或新增一个 bridge skill 时，优先遵循以下约定：

1. 保留来源 skill 的独立目录和名称
2. 核心保留 `SKILL.md`、`README.md`、`config.yaml`、`scripts/`；若来源 skill 已有 `plans/`、`tests/`、`docs/`，也应一并保留
3. 排除 `.DS_Store`、`__pycache__/`、`*.pyc` 等缓存产物
4. 在根级 `README.md` 或 `skills/README.md` 中登记新 skill 的用途与边界
5. 在 `CHANGELOG.md` 中记录迁移、重构或新增情况

## 变更记录规范

凡是以下变更，都必须记录到 `CHANGELOG.md`：

- 新增、迁移、删除或重命名 skill
- 修改仓库级说明文件：`AGENTS.md`、`CLAUDE.md`、`README.md`
- 修改影响多个 skill 的结构约定或共享规则
- 调整 `.gitignore`、目录布局或其他仓库级配置

## Codex CLI 特定说明

- 引用文件时使用可点击路径，并尽量附带起始行号
- 修改前先读取足够上下文，再做成组修改
- 如果只是迁移和初始化，不要假装完成了上游 API 的在线联调；未验证就明确说明未验证

## 有机更新原则

- 当新增第三个、第四个产品 skill 时，优先扩展当前结构，而不是推倒重来
- 若仓库开始出现稳定共享层，再逐步抽取公共文档或工具
- 更新仓库级规则时，同时检查 `README.md`、`skills/README.md` 与 `CHANGELOG.md` 是否需要同步
